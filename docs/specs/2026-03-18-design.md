# alpha-claude-telegram-ask-user -- Design Spec
Date: 2026-03-18
Status: v4 (post spec-review round 3)

## 1. Purpose

A standalone npm package that bridges the Claude Agent SDK `ask_user` MCP tool with Telegram inline keyboards. When a Claude agent calls `ask_user`, the user receives a Telegram message with clickable buttons and the agent waits for the answer before continuing.

No other open source project implements this bridge.

## 2. Scope

**In scope:**
- MCP server exposing the `ask_user` tool (Claude Agent SDK)
- Telegram inline keyboard builder (single-select and multi-select)
- ForceReply handler for free-text answers ("Autre" option)
- Callback query handler that resolves the pending promise
- In-memory storage (default) with optional async storage adapter interface
- Configurable timeout (default 5 minutes)
- grammY adapter (included)
- Telegraf adapter (included)

**Out of scope:**
- Discord, Slack, or other transports
- Persistent storage implementation (interface provided, implementation delegated to user)
- Bot creation or management (user brings their own bot)

## 3. Architecture

```
User code
  |
  +--> const ui = createTelegramUiServer(adapter, chatId, options?)
  |      ui.server  <-- pass to mcpServers in Claude query
  |      ui.handleCallbackQuery(callbackData, callbackQueryId)  <-- call from bot
  |      ui.handleForceReplyMessage(replyToMessageId, text)     <-- call from bot
  |
```

The cleanup interval is started internally by `createTelegramUiServer` and stopped via `ui.destroy()`.

### 3.1 TelegramAdapter interface

Abstracts grammY, Telegraf, or any future transport. Uses a framework-agnostic keyboard type.

```ts
// Framework-agnostic inline keyboard markup
interface InlineButton {
  text: string
  callback_data: string
}

type InlineKeyboardMarkup = InlineButton[][]  // rows of buttons

interface TelegramAdapter {
  sendMessage(
    chatId: string,
    text: string,
    keyboard?: InlineKeyboardMarkup
  ): Promise<{ messageId: number }>

  editMessage(
    chatId: string,
    messageId: number,
    text: string,
    keyboard?: InlineKeyboardMarkup  // undefined = remove keyboard
  ): Promise<void>

  answerCallbackQuery(
    callbackQueryId: string,
    text?: string  // optional toast notification
  ): Promise<void>

  sendForceReply(
    chatId: string,
    promptText: string
  ): Promise<{ messageId: number }>
}
```

Included adapters:
- `createGrammyAdapter(bot: Bot)` -- wraps grammY
- `createTelegrafAdapter(bot: Telegraf)` -- wraps Telegraf

### 3.2 Storage interface

All methods are async to support both in-memory and network-backed stores.

```ts
interface QuestionStore {
  set(id: string, question: PendingQuestion): Promise<void>
  get(id: string): Promise<PendingQuestion | undefined>
  delete(id: string): Promise<void>
  getExpired(now: number): Promise<PendingQuestion[]>
}
```

Default: `InMemoryStore` (built-in, no external dependencies).
Optional: user injects their own (SQLite, Redis, etc.) -- all async.

### 3.3 Module-level singletons and multi-instance behavior

`pendingResolvers` (`Map<questionId, resolver>`) and `pendingForceReplies` (`Map<messageId, questionId>`) are module-level singletons.

**Constraint: call `createTelegramUiServer` only once per process.**

Calling it multiple times is unsafe: instance A's `handleCallbackQuery` can accidentally resolve instance B's pending questions because both instances share the same resolver map. There is no routing by chatId at the resolver level.

`chatId` is a single `string`. If you need to handle multiple chat IDs, route them externally and pass a single authoritative chatId to the factory. Multi-instance support is out of scope for v1.

### 3.4 MCP Server

Created via `createTelegramUiServer(adapter, chatId, options?)`.

Options:
```ts
{
  timeoutMs?: number      // default: 300_000 (5 min)
  store?: QuestionStore   // default: new InMemoryStore()
}
```

Returns:
```ts
// McpServer is the return type of createSdkMcpServer() from @anthropic-ai/claude-agent-sdk
{
  server: McpServer               // pass to mcpServers in Claude query
  handleCallbackQuery(           // call from bot's callback_query handler
    callbackData: string,
    callbackQueryId: string
  ): Promise<void>
  handleForceReplyMessage(       // call from bot's message handler
    replyToMessageId: number,
    answerText: string
  ): Promise<void>
  destroy(): void                // stops internal cleanup interval
}
```

Exposes one tool: `ask_user` with the following JSON Schema:
- `question` (string, required) -- the question text
- `options` (array, required, `minItems: 2`, `maxItems: 4`) -- `{ label: string, description: string }`
- `header` (string, optional) -- short label shown in brackets
- `multi_select` (boolean, optional, default false)

If `options` contains fewer than 2 or more than 4 items, the MCP tool returns a validation error string to the agent: `"ask_user requires between 2 and 4 options."` No Telegram message is sent.

Return values:
- Single-select: the chosen label string (e.g. `"Option A"`)
- Multi-select: comma-separated string of chosen labels (e.g. `"Option A, Option C"`)
- Timeout: `"User did not respond within X minutes. Proceed with best judgment."`
- Error: `"Failed to reach user. Proceed with best judgment."`

### 3.5 Multi-select edge cases

- **Valider pressed with zero items selected**: `answerCallbackQuery` is called with toast `"Select at least one option"` and the keyboard stays open. The promise is not resolved.
- **Valider pressed with items selected**: resolves with comma-separated labels string.

### 3.6 Pending resolvers

Module-level `Map<questionId, { resolve: (answer: string) => void }>` shared between:
- `createTelegramUiServer` (writes resolvers when `ask_user` is called)
- `handleCallbackQuery` (resolves them on user tap)
- internal cleanup interval (resolves with timeout sentinel `__timeout__`)

### 3.7 ForceReply ("Autre")

Module-level `Map<messageId, questionId>` (`pendingForceReplies`) managed internally. The `handleForceReplyMessage(replyToMessageId, text)` method does the `messageId -> questionId` lookup internally. The user does not need to access `pendingForceReplies` directly.

### 3.8 Timeout and expired question cleanup

**Timeout path (immediate):** when `ask_user` is called, a `setTimeout` is set for `timeoutMs`. When it fires, the Telegram message is edited immediately to show expiry, and the promise resolves with `__timeout__`. The `server.ts` tool handler translates `__timeout__` to the human-readable timeout string before returning to Claude.

**Cleanup interval (safety net):** `setInterval` runs every 60s to catch any questions whose `setTimeout` was not honored (e.g., process pause, missed timer). It calls the same edit + resolve logic for any expired unanswered questions.

`destroy()` stops the cleanup interval and clears both `pendingResolvers` and `pendingForceReplies`. Any in-flight questions at destroy time are silently dropped (no Telegram edit, no resolution). This is acceptable because `destroy()` is intended for process shutdown.

## 6. Error handling

| Scenario | Behavior |
|---|---|
| Telegram API fails on sendMessage | Return graceful error string to Claude; delete question from store |
| Timeout | Edit Telegram message to show expiry; resolve with timeout message |
| Double-tap (already answered) | `answerCallbackQuery` silently; no-op |
| Unknown questionId in callback | Log warning; `answerCallbackQuery` silently |
| Valider with zero selections | Toast "Select at least one option"; keep keyboard open |
| ForceReply replyToMessageId not found | Silently ignore |

## 7. Success criteria

- `npm install alpha-claude-telegram-ask-user` with no mandatory peer deps beyond Anthropic SDK
- grammY example runs end-to-end: agent asks question, user taps button, agent receives answer
- Telegraf example runs end-to-end
- Custom store example shows async SQLite injection
- No circular imports
- TypeScript strict mode passes
- `ui.destroy()` stops the cleanup interval cleanly
